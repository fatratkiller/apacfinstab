## 🧠 产品本质定义 (2026-02-26 确立)

### L1.5 情境层的具体含义

```
我们卡在L1.5，不碰L2！

L1 数据层: PolicyPedia (监管数据MCP) → 所有Agent
L1.5 情境层: Regulatory Context API → AI产品公司 ← 我们在这里
L2 判断层: 人类合规团队 → 我们不碰
```

### Value Prop

**旧版 ❌：**
> "合规检查API，帮Agent判断是否合规"

**新版 ✅：**
> "监管情境感知API，让Agent知道自己在什么环境里运行，但不替你做判断"

### 核心设计原则

| 原则 | 说明 |
|------|------|
| **不返回 true/false** | 返回 confidence + flags + references |
| **不做判断** | 标记风险点，让人类决策 |
| **苏格拉底式** | 问出好问题，不给答案 |
| **责任边界清晰** | 风险发现 → AI；最终判断 → 人类 |

### API设计方向

```python
result = check_compliance_context(
    action="launch_staking_product",
    jurisdiction="HK",
    entity_type="VASP"
)

# 返回:
{
    "confidence": 0.73,           # 基于规则明确程度
    "status": "NEEDS_REVIEW",     # CLEAR/NEEDS_REVIEW/HIGH_RISK/UNKNOWN
    "flags": [                    # 苏格拉底式提问
        "SFC可能视为regulated activity — 你确认过业务定义吗？",
        "需确认是否符合Type 1牌照范围"
    ],
    "references": ["SFC FAQ Q12", "VASP Guideline 4.3"],
    "suggested_actions": ["Review with legal counsel"]
}
```

### 为什么AI产品公司会买？

```
想做金融相关Agent → 担心合规风险 → 不敢上线
                    ↓
        需要一个"合规护栏层"
                    ↓
        不是替他们判断，是告诉他们"这里有风险"
```

**类比：**
- 自动驾驶的"接管提醒" — 不是替你开，是告诉你什么时候该接管
- IDE的"代码警告" — 不是替你修，是告诉你哪里可能有问题

## 🔮 Predictive Compliance (2026-02-27 确立)

### 产品能力分层

```
Level 1: "告诉你规则" → 低风险，低价值（竞品都做）
Level 2: "告诉你规则可能变" → 中风险，高价值 ← 我们在这里
Level 3: "告诉你应该怎么做" → 高风险，我们不做
```

**苏格拉底式预测 = Level 2**
> 不给答案，给洞察 + 问题
> 例："SFC稳定币框架可能在Q2收紧，你考虑过时间窗口吗？"

### 预测标准定义

| 标准 | 定义 |
|------|------|
| **命中** | 方向正确 + 时间±3个月 |
| **部分命中** | 方向正确，时间偏差>3个月 |
| **未命中** | 方向错误 或 事件未发生 |

### 预测范围（扩大sample size）

- ✅ 新规出台
- ✅ 执法行动
- ✅ 牌照批准时间线
- ✅ 政策声明措辞变化
- ✅ 监管者人事变动影响

### 预测结构化格式

```json
{
  "prediction_id": "HK-SFC-2026-001",
  "jurisdiction": "Hong Kong",
  "topic": "稳定币发行人牌照框架",
  "prediction": "将增加储备金审计频率要求",
  "confidence": "medium",
  "predicted_timeframe": "2026-Q2 to 2026-Q3",
  "reasoning_chain": [
    "HKMA 2月会议纪要措辞变化",
    "新加坡MAS已实施类似要求",
    "SFC官员公开发言信号"
  ],
  "created_at": "2026-02-27",
  "verified_at": null,
  "outcome": null
}
```

### 预测Track Record机制

```
Phase 1 (现在): 开始记录预测，内部积累
    ↓
Phase 2 (6个月后): 有track record后，小范围公开
    ↓
Phase 3 (1年后): 预测准确率成为核心卖点
```

**关键原则：失败预测也公开 + post-mortem分析**

这是最大的差异化 — 传统合规供应商永远不会承认错误，我们公开post-mortem，这本身就是品牌。

### 护城河来源

| 护城河 | 说明 |
|--------|------|
| ❌ 数据 | 监管数据是公开的 |
| ❌ 规则引擎 | 技术可复制 |
| ✅ 因果图谱深度 | 需要专家+时间积累 |
| ✅ 预测Track Record | 需要时间验证 |
| ✅ 生态位嵌入 | MCP生态先发优势 |

**真正的护城河 = 被AI Agent依赖的程度**
